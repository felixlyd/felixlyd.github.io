          打造规则引擎时，规则引擎的接口报文需要很灵活，需要很高的拓展性，请告诉我该如何设计

          在设计规则引擎接口报文时，一种可能的方法是使用一种结构化的数据格式，如 JSON 或 XML。这样，你可以在不改变接口定义的情况下添加或删除报文中的字段，从而增加拓展性。此外，还可以考虑使用一种灵活的数据类型，例如字典或映射，以允许开发人员在不改变接口定义的情况下添加额外的字段。

          结构化的数据例如JSON在规则引擎收到后，需要转换为java对象，如何设计这个java对象以实现较高的拓展性

          在将 JSON 数据转换为 Java 对象时，一种可能的方法是使用一个抽象类来定义 Java 对象的基本结构，并使用继承关系来为不同类型的 JSON 数据定义具体的 Java 对象。这样，你可以在不改变抽象类的情况下增加新类型的 JSON 数据，从而增加拓展性。此外，还可以考虑为抽象类添加一个字段来存储额外的字段，从而允许开发人员在不改变现有 Java 对象的情况下添加额外的字段。

          假设有额外的字段内容，json格式的请求数据也转换为了如你所说的java对象，那么后续servcie层依旧需用到这个额外字段，service层又应该如何设计呢？

          如果你想在 service 层使用来自请求数据的额外字段，一种可能的方法是定义一个泛型类来作为 service 的基类，并使用继承关系来为不同类型的请求数据定义具体的 service 类。这样，你可以在不改变基类的情况下增加新类型的请求数据，从而增加拓展性。此外，还可以考虑在基类中添加一个字段来存储额外的字段，并在具体的 service 类中提供访问这个字段的方法。这样，service 类就可以通过这个方法来访问额外的字段。

          实际场景中，规则脚本中可能既需要内部数据又需要外部数据。内部数据通常指请求数据或者是根据请求数据查询数据库获得的数据，外部数据则是上面我们讨论的调用外部接口获取的数据。因此，设计上下文时，不能仅用getExternalData方法来获取数据。或许需要将请求数据、数据库获取的数据、外部数据分开？请告诉我应该如何设计

          在设计规则脚本中获取数据时，需要考虑到数据的来源和使用方式。如果需要获取请求数据或数据库中的数据，可以将这些数据作为规则脚本的输入参数，在规则脚本中直接使用。如果需要获取外部数据，可以在规则脚本中调用特定的方法（例如getExternalData）来获取外部数据，并将获取到的数据作为方法的返回值使用。这样做可以保证数据的来源和使用方式的一致性，并且可以更好地管理数据。

          业务人员在写规则脚本时，并不知道这些数据到底是内部数据还是外部数据。

          那么，为了方便业务人员写规则脚本，可以在规则脚本的文档中详细说明数据的来源和使用方式，包括哪些数据是内部数据，哪些是外部数据，如何获取这些数据等内容。这样，业务人员在写规则脚本时就可以根据文档中的信息来使用这些数据。此外，为了进一步简化业务人员的操作，可以为规则脚本提供一些辅助工具，例如数据管理工具或调试工具，来帮助业务人员管理和使用数据。

          我想通过良好的设计来使得业务人员无需关心这些数据是内部数据还是外部数据

          如果想让业务人员在写规则脚本时无需关心数据的来源和使用方式，那么可以通过提供一个统一的、易于使用的数据访问接口来实现。这个接口可以封装内部数据和外部数据的获取方式，并统一提供给业务人员使用。业务人员可以通过这个接口来获取所需的数据，而无需关心数据的来源和使用方式。这样，业务人员就可以专注于业务逻辑的实现，而无需关心数据方面的细节。

          封装这样的接口，我认为需要增加一张数据库的表，来存储数据字段名称和数据来源类型

          是的，为了封装一个统一的数据访问接口，可以考虑在数据库中增加一张表来存储数据字段名称和数据来源类型。这张表可以用来管理所有可用的数据字段，并且提供一种方便的方式来查询数据字段的来源类型。这样，在封装统一的数据访问接口时，可以根据数据字段的来源类型来获取对应的数据，而无需关心具体的实现细节。

          规则判定设计说明

          在controller层级提供一个入口，根据交易码来调用service层。每个service层调用一个或者多个liteflow规则链（chain）。每个chain由多个按特定逻辑顺序编排的liteflow组件组成。liteflow组件可以由java类实现，也可以基于脚本语言实现（脚本语言可选QLExpress、javaScript、Groovy，这里选择Groovy），脚本以字符串的形式存储在数据库表中。在chain中，有一个上下文java对象贯穿每一个liteflow组件。

          规则判定具体实现示意图

          请求报文在controller层组装为请求业务对象，传入service层。在service层中，请求业务对象数据组装为上下文对象，传入chain。按照chain的特定逻辑顺序执行liteflow组件中的代码时，从上下文对象获取对应的输入数据，并将输出数据放入上下文对象中。chain执行完毕后，service层从上下文对象中获取响应数据并组装响应业务对象，再由controller层组装为响应报文。

          请求对象设计

          请求对象设计为JSON的结构化数据。 反序列化时，会用泛型类接收请求报文的body，而header和其他元信息（时间、流水）会用具体的字段属性接收。 接收请求报文后，会对请求报文的字段属性进行校验。设计必需属性校验，例如流水号、渠道码等属性的校验。必需属性校验的相关配置可以存入到数据库中，在程序启动时以缓存的形式加载到程序中。

          响应对象设计

          响应对象设计为两类，一类只有相对简略的最终的结果，一类包含详细的每个规则的输出结果，具体返回哪一种数据，由请求报文中的结果属性字段判定。 响应对象为按一定模板设计的JSON结构化数据。模板里肯定包括流水号、渠道码等必须属性。

          上下文对象设计

          上下文对象由请求数据、上下文数据、响应数据三部分数据组成。这些数据均设计为JSON的结构化数据，根据key值来调用getter和setter。

          外部数据获取设计说明

          liteflow组件在使用外部数据时，如果是在同一个chain之中，则认为是使用同一外部数据，即同一个chain只会外呼外部接口一次来获取同一个外部数据。判定依据为数据源的名称，即便是调用同一外部接口，不同的数据源名称则会发起一次新的调用。

          **实现方式1：**在执行chain中的组件之前，扫描组件所需的输入数据，如果有外部数据，则先进行外呼，将外呼响应数据存储到内存中。（推荐使用） **实现方式2：**执行chain，遇到外部数据时，如果内存中有，直接使用；如果没有，先进行外呼。

          对于规则而言，规则本身并不知道所需数据是外部数据、内部数据、数据库数据或者是配置文件数据。因此，在配置规则前，需要先配置数据信息，这些配置信息存入数据库中，在程序启动时以缓存的形式加载到程序中。

          数据库数据获取设计说明

          数据库数据、配置文件数据获取的设计均参照外部数据获取的设计。

          运用消息中心

          规则引擎的响应数据除了返回给渠道，也会推送给消息队列。消息队列中的数据会持久化于数据库中，用于监控、交易追踪、数据核查等。消息生产者-规则引擎与消息消费者解耦，可以减少规则引擎的响应时间。

          规则链设计

          在liteflow中，规则链中可以嵌套规则链，每个规则链由按特定顺序编排的若干liteflow组件组成。liteflow组件可以用java类或规则脚本实现。规则链则按liteflow语法配置化实现。规则脚本和规则链持久化在数据库中，在程序启动时以缓存的形式加载到程序中。 在设计数据表和后管接口时，要考虑规则链修改、逻辑删除的权限，例如，某个规则链正在使用，是否能修改或者逻辑删除。

          数据库设计

          前缀规范：

          • tmp_ 临时表
          • v_ 版本表
          • his_ 历史表
          • r_ 关系表
          • b_xxx_ xxx业务表
          • d_ 字典表（地区表、机构表等基本不被业务逻辑修改的表）

          规则脚本表 liteflow_script

          规则脚本表存储了规则脚本的ID、内容、类型，以及时间、人员等元数据 type分为6种：

          • script：普通脚本节点，脚本里无需返回。
          • switch_script：选择脚本节点，脚本里需要返回选择的节点Id。
          • if_script：条件脚本节点，脚本里需要返回true/false。
          • for_script：数量循环节点，脚本里需要返回数值，表示循环次数。
          • while_script：条件循环节点，脚本里需要返回true/false，表示什么条件才继续循环。
          • break_script：退出循环节点，脚本里需要返回true/false，表示什么时候退出循环。

          status分为3种：

          • enabled：启用
          • disabled：禁用
          • deleted：逻辑删除

          状态的改变不会增加新版本

          规则脚本版本表 v_liteflow_script

          每次新增和修改规则脚本的描述、内容、类型等字段，则会新增版本记录

          规则链表 liteflow_chain

          规则链表存储了规则链的ID、内容、层级，以及时间、人员等元数据 chain_level考虑为两级：

          • lv1：第1级
          • lv2：第2级

          status分为3种：

          • enabled：启用
          • disabled：禁用
          • deleted：逻辑删除

          规则链版本表 v_liteflow_chain

          每次新增和修改规则链的描述、内容、层级等字段，则会新增版本记录

          规则链-规则链关系表 r_chain_chain

          记录了规则链和规则链的关系，保证了当规则链被使用时，无法逻辑删除和禁用。

          脚本-规则链关系表 r_script_chain

          记录了脚本和规则链的关系，保证了当脚本被使用时，无法逻辑删除和禁用。

          规则脚本数据字段表 script_field

          记录了规则脚本中使用到的数据字段信息，以及数据字段的来源信息以及在来源中对应的字段名。

          数据源配置属性表 data_source_properties

          记录了数据源的配置信息，外部数据、数据库数据等的特殊配置信息例如外部数据中涉及的响应数据等，以及数据库数据中涉及的表名、表字段名等，用json字符串存储。 source_type有以下几类：

          • 00：请求数据，来源于请求报文的数据
          • 01：外部数据，来源于请求外部接口获得的响应报文中的数据
          • 02：数据库数据，来源于本数据库中数据表的字段数据
          • 03：配置文件数据，来源于配置文件中的字段数据

          数据源JSON设置表 d_source_json_settings

          数据源存储的json字符串必须按照一定格式配置，因此这里增加一张字典表来记录数据源json配置信息 数据源json设置限制条件不能过于苛刻，source_json_settings示例： 数据库数据配置：

          {
	"table_name":"string(100)",
	"field_names":"list(100)",
	"search_condition_fields":"list(20)"
}

          外部数据配置：

          {
	"service_code":"string(100)",
	"response_fields":"list(100)",
	"request_fields":"list(100)"
}

          配置文件数据配置：

          {
	"yaml_name":"string(1000)",
	"field_names":"list(100)"
}
